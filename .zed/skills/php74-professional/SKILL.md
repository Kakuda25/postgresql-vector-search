# PHP 7.4 プロフェッショナル

PHP 7.4 の言語仕様とプロ品質を満たすための指針。コードの作成・レビュー・リファクタ時に適用する。

## 基本方針

- **型の明示**: プロパティ・引数・戻り値に型宣言を付ける。PHP 7.4 の typed properties を積極的に使う。
- **スタイル**: PSR-1 / PSR-12 に準拠する。プロジェクトで別ルールがある場合はそれに従う。
- **読みやすさ**: 意図が伝わる命名、適切な関数・クラス分割、過剰なコメントは避ける。

## 型宣言（PHP 7.4）

- クラスプロパティに型を宣言する。未初期化アクセスは `Error` になるため、コンストラクタで必ず代入するか、nullable にする。
- 使用可能な型: `int`, `float`, `string`, `bool`, `array`, `object`, `iterable`, `?Type`（nullable）, クラス名・インターフェース名。
- 静的プロパティにも型宣言を付ける。未使用の型（`void` は戻り値のみ）はプロパティに使わない。

```php
class User {
    public int $id;
    public string $name;
    public ?string $email = null;
}
```

## アロー関数（PHP 7.4）

- 単一式のクロージャには `fn (args) => expr` を使う。`use` は不要で、親スコープを by-value で自動取得する。
- 複数行や複文が必要な場合は従来の `function () use (...) { }` を使う。
- 引数・戻り値の型ヒントは付けてよい。

```php
$factor = 2;
$result = array_map(fn (int $n): int => $n * $factor, [1, 2, 3]);
```

## 演算子・構文

- Null 合体代入 `??=` で「未定義・null のときだけ代入」を簡潔に書く。
- 配列の展開に `...` を使う（PHP 7.4 で配列リテラル内で利用可能）。
- 数値リテラルの可読性のために `_` を利用してよい（例: `1_000_000`）。
- 三項演算子は括弧で優先順位を明示する。左結合の曖昧な書き方は非推奨のため避ける。

## エラーと例外

- 実行時エラーは `Throwable`（`Error` / `Exception`）で扱う。`Error` も捕捉してログや適切なレスポンスに変換する場合は `catch (Throwable $e)` を検討する。
- 捕捉する例外は可能な限り具体的な型で catch する。`catch (Exception $e)` のみで握りつぶさない。
- ユーザー向けメッセージとログ用（スタック含む）を分け、本番ではスタックをログにのみ出力する。
- `__toString()` 内で例外を投げることが PHP 7.4 で許容されるが、なるべく例外を投げない実装にする。

## セキュリティ

- SQL は必ずプリペアドステートメント（PDO / mysqli）で実行する。ユーザー入力を文字列連結でクエリに含めない。
- HTML 出力には `htmlspecialchars($str, ENT_QUOTES, 'UTF-8')` を必ず適用する。属性値もエスケープする。
- フォームには CSRF トークンを付け、POST 時に検証する。
- ファイルパス・コマンドライン引数はユーザー入力をそのまま使わず、許可リストや正規化で検証する。
- 機密情報はソースにハードコードしない。環境変数や設定ファイル（リポジトリ外）で扱う。

## 入力とバリデーション

- 外部入力（`$_GET`, `$_POST`, `$_COOKIE`, ヘッダ）は信頼しない。型・範囲・形式を検証してからビジネスロジックに渡す。
- フィルタ系: `filter_var`（FILTER_VALIDATE_* / FILTER_SANITIZE_*）を用途に合わせて使う。サニタイズだけに頼らず、検証と組み合わせる。

## オブジェクトと API

- 不変にしたい値は読み取り専用プロパティや `readonly` 的運用で表現する（PHP 8.1 以前では `readonly` はないため、コンストラクタ代入のみにする等）。
- プロパティの存在は `property_exists` や型付きプロパティで未初期化を避ける設計にする。
- インターフェースで契約を明示し、依存性の注入でテストしやすくする。

## パフォーマンスとメモリ

- ループ内の不要なオブジェクト生成・重い処理を避ける。大きな配列はイテレータやジェネレータの利用を検討する。
- 不要なグローバルや静的変数は避け、スコープを狭める。
- 拡張・フレームワークのベストプラクティス（オートロード、オプキャッシュ等）に従う。

## 禁止・非推奨

- `mysql_*` は使用しない。PDO または mysqli を使う。
- `eval()` / `create_function()` は使わない。ユーザー入力を渡さない。
- エラー抑制 `@` は例外処理や戻り値チェックで代替する。やむを得ず使う場合はコメントで理由を残す。
- 左結合の曖昧な三項演算子（PHP 7.4 で非推奨）は括弧で明確にする。
- 動的プロパティの乱用は避け、型付きプロパティで意図を明示する。

## レビュー時のチェック

- [ ] 全ての SQL がプリペアドステートメントか
- [ ] HTML 出力のエスケープ漏れがないか
- [ ] 型宣言（プロパティ・引数・戻り値）が適切か
- [ ] 例外・エラーの捕捉とログ出力が適切か
- [ ] 外部入力の検証と許可範囲の確認
- [ ] ループ・再帰の終了条件とリソース解放
- [ ] 依存ライブラリのバージョンと既知の脆弱性（Composer 等）
